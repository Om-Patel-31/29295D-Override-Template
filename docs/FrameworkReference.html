VEX V5 Motion Framework

Stable

This reference defines the motion API contract, core control architecture, autonomous selection flow, and purpose of each project file. GPS coordinates are treated as the field truth source for odometry, and the distance sensor is used by drive PID for distance-to-target control with configurable offset.

Quick Use

void pre_auton() {
  configureHardware();
  initializeFramework();
  InertialSensor.calibrate();
  while (InertialSensor.isCalibrating()) {
    wait(50, vex::msec);
  }
}

void autonomous() {
  resetAutonomousState(0.0, 0.0, InertialSensor.heading(vex::deg));

  driveDistance(Direction::Forward, 24.0);
  turnToHeading(90.0);
  driveToCoordinates(36.0, 18.0, 90.0);
}

Complete Function Usage Snippet

This example is intentionally comprehensive and shows one call-site for every custom framework function.

#include "vex.h"  // Pulls in the VEX SDK types and runtime used by every example below.
#include "framework/AutonManager.h"  // Exposes auton selection and launch helpers so the demo can show the full public API.
#include "framework/ChassisController.h"  // Provides the drivetrain controller class used in the internal-layer example.
#include "framework/MotionAPI.h"  // Declares the project motion entry points used by student code.
#include "framework/Odometry.h"  // Supplies the Pose2D type and odometry helper functions for field position logic.
#include "framework/PID.h"  // Gives access to the reusable PID controller used by drive and turn loops.
#include "framework/RobotConfig.h"  // Brings in hardware objects like motors, sensors, and the Brain display.
#include "framework/Tuning.h"  // Supplies the tunable constants used to keep the example aligned with project defaults.

using namespace framework;  // Lets the demo call framework APIs without repeating the namespace everywhere.

void allFunctionUsageDemo() {  // Groups every framework call into one readable reference example.
  // Public setup and auton-management API.
  configureHardware();  // Applies the motor stop policy before motion starts so drivetrain behavior stays consistent.
  initializeFramework();  // Constructs the motion stack so the API can talk to hardware and odometry.
  initializeAutonProgram();  // Resets auton state so the selector starts from a predictable default.
  selectAutonWithController();  // Lets the driver choose the auton routine on the controller before a match begins.
  const char* autonName = selectedAutonName();  // Reads the selected auton name so the example shows how to inspect it.
  if (autonName != nullptr) {  // Guards the display call so the pointer is used as real data instead of being discarded.
    Brain.Screen.print("Selected auton: %s", autonName);  // Shows the selected auton on the Brain for operator confirmation.
  }  // Ends the display guard after the name has been used.

  resetAutonomousState(0.0, 0.0, InertialSensor.heading(vex::deg));  // Starts auton from a known field pose and current heading.

  driveDistance(Direction::Forward, 24.0);  // Demonstrates a simple forward distance move with the default motion tuning.
  driveDistance(Direction::Reverse, 12.0, 50.0, 1800, tuning::kDrivePID);  // Shows the overload that adds speed, timeout, and explicit PID tuning.
  turnToHeading(90.0);  // Commands an in-place turn to a specific field heading.
  turnToHeading(180.0, 1600, tuning::kTurnPID);  // Demonstrates the timeout and PID-tuned turn overload.
  driveToCoordinates(36.0, 18.0, 90.0);  // Drives to a known field point while finishing at a target heading.
  driveToCoordinates(24.0, 48.0, 180.0, 55.0, 2500, tuning::kSDriveTune);  // Shows the full waypoint routine with custom velocity, timeout, and tuning.

  Pose2D apiPose = getPose();  // Reads the fused global pose so telemetry and decisions can use live position data.
  Brain.Screen.print("API pose: x=%.1f y=%.1f h=%.1f", apiPose.x, apiPose.y, apiPose.headingDeg);  // Uses the pose values so the sample stays discard-free.

  runSelectedAuton();  // Runs whichever auton the driver selected so the demo covers the competition entry point.

  // Internal classes: OdometryTracker function usage.
  OdometryTracker odom(LeftRotationA, LeftRotationB, RightRotationA, RightRotationB,  // Binds the wheel trackers that measure chassis travel.
                       InertialSensor, GpsSensor);  // Supplies heading and GPS data so the odometry tracker can fuse a field pose.
  odom.rst(0.0, 0.0, 0.0);  // Zeroes the tracker so the example starts from a known origin.
  odom.set(5.0, 10.0, 45.0);  // Manually sets a pose to show how to correct or seed odometry.
  odom.upd();  // Refreshes the pose estimate from the live sensors.
  Pose2D odomPose = odom.getPose();  // Captures the fused pose for later math and display.
  double wrappedHeading = OdometryTracker::norm(370.0);  // Normalizes an angle to show heading wrapping behavior.
  double turnError = OdometryTracker::herr(90.0, odomPose.headingDeg);  // Computes the shortest signed turn error from the current pose.
  Brain.Screen.print("Odom pose: x=%.1f y=%.1f h=%.1f wrap=%.1f err=%.1f",  // Displays all odometry outputs so each value is used in the example.
                     odomPose.x, odomPose.y, odomPose.headingDeg, wrappedHeading, turnError);  // Shows how the tracker supports field navigation math.

  // Internal classes: PIDController function usage.
  PIDController pid(tuning::kDrivePID);  // Creates a PID controller with the default drive gains used by the project.
  pid.set(tuning::kTurnPID);  // Reconfigures the same controller to show how tuning can be swapped at runtime.
  pid.rst();  // Clears accumulated state so the next control step starts cleanly.
  double pidCmd = pid.calc(8.0, tuning::kControlLoopDtSec);  // Produces a command from the current error and control-loop timing.
  bool pidAtTarget = pid.ok(0.4);  // Checks whether the current error is inside the tolerance window.
  double pTerm = pid.p();  // Reads the proportional contribution for tuning and debugging.
  double iTerm = pid.i();  // Reads the integral contribution so windup can be inspected.
  double dTerm = pid.d();  // Reads the derivative contribution to understand damping behavior.
  Brain.Screen.print("PID cmd=%.1f ok=%d P=%.1f I=%.1f D=%.1f", pidCmd, pidAtTarget, pTerm, iTerm, dTerm);  // Uses every PID output so nothing is discarded.

  // Internal classes: ChassisController function usage.
  ChassisController chassis(LeftFrontMotor, LeftBackMotor, RightFrontMotor, RightBackMotor, odom);  // Binds the drivetrain motors to the odometry tracker.
  chassis.driveDistance(18.0, 60.0, 2000, tuning::kDrivePID);  // Drives a fixed distance using the lower-level chassis controller.
  chassis.turnToHeading(135.0, 1500, tuning::kTurnPID);  // Commands a precise heading change using the controller layer.
  chassis.driveToCoordinates(30.0, 30.0, 180.0, 55.0, 2500, tuning::kSDriveTune);  // Shows the full GPS-waypoint routine through the chassis wrapper.
  Pose2D chassisPose = chassis.getPose();  // Reads the controller-owned pose for live debugging or telemetry.
  Brain.Screen.print("Chassis pose: x=%.1f y=%.1f h=%.1f", chassisPose.x, chassisPose.y, chassisPose.headingDeg);  // Uses the chassis pose so the example never needs a discard cast.
  chassis.stop();  // Stops the drivetrain so the example ends in a safe state.
}

Internal Control Layers

The control stack is layered so students can use the public API while advanced users tune the internals when needed.

FileFunctionPurposeUse case
include/framework/PID.hset()Applies a new PID tune and resets controller memory.Swap gains between movement phases without constructing a new controller object.
include/framework/PID.hrst()Clears integral, derivative history, and term outputs.Start a fresh control action so old accumulated error does not affect new commands.
include/framework/PID.hcalc()Runs one PID step and returns output command.Called each loop tick to convert current error into motor command.
include/framework/PID.hok()Checks if error is inside tolerance window.End a loop when target is reached instead of relying only on timeout.
include/framework/Odometry.hrst()Resets wheel tracking and sets pose.Initialize autonomous from known field coordinates and heading.
include/framework/Odometry.hset()Directly sets fused pose values.Manually correct pose estimate after special field setup or debugging.
include/framework/Odometry.hupd()Integrates sensors to refresh fused pose.Run every control cycle before path decisions or telemetry reads.
include/framework/Odometry.hgetPose()Returns current fused pose state.Feed navigation routines and on-screen debug output.
include/framework/Odometry.hnorm()Wraps heading angle into [0, 360) degrees.Normalize any heading math before comparing or displaying angles.
include/framework/Odometry.hherr()Computes shortest signed heading error.Turn control loops where robot should rotate the shortest direction to target.
include/framework/ChassisController.hstop()Stops all drivetrain motors.Safely end motion commands or halt robot on timeout.
include/framework/ChassisController.hdriveDistance()Closed-loop straight-line drive command.Move to a desired distance target using PID, with front distance sensor + back offset when valid and odometry fallback otherwise.
include/framework/ChassisController.hturnToHeading()Closed-loop turn to heading command.Align robot to target orientation between path segments.
include/framework/ChassisController.hdriveToCoordinates()Closed-loop translation + heading blend command.Follow a field-space waypoint and settle on a final heading in one action.
include/framework/ChassisController.hgetPose()Exposes odometry pose through chassis layer.Inspect controller-owned odometry pose during debugging.

PID behavior

output = P + I + D P = kP * error I = kI * integral(error over time), clamped to prevent windup D = kD * change in error / dt

Odometry behavior

Forward movement comes from the two left and two right rotation sensors. Heading comes from the inertial sensor. GPS coordinates gently correct field x/y/heading. Distance sensor is not fused into odometry.

Swing turns create controlled arcs by reducing inside-wheel output when heading error is large. If one axis reaches target first, the command continues until the full GPS coordinate error vector is inside tolerance, then final heading is settled.
